# OCR Platform Manifest and Shard Queue Architecture

Date: 2026-05-25

## Purpose

This document defines the next-stage execution architecture for very large OCR
jobs, such as millions or tens of millions of PDFs stored on a shared disk and
processed by multiple execution servers.

The current MVP runs one directory-level job on one assigned execution server.
That shape is useful for validation, but it does not scale well when a single
input folder should be shared by 10 or more execution servers.

The target model is:

- the user specifies one input folder or an existing manifest;
- the control plane creates or accepts a manifest snapshot;
- the platform splits the manifest into many bounded shards;
- execution agents dynamically claim shards with leases;
- agents read PDFs from shared storage, call the load-balanced OCR URL, write
  outputs back to shared storage, and report shard-level progress;
- the UI stays job-level by default and never lists millions of PDFs.

## Design Principles

- A manifest is a job input snapshot, not a live view of the folder forever.
- Nested folders are preserved through `relative_path`, but scheduling is not
  based on folder boundaries.
- The control plane or a dedicated scanner scans once. Execution agents should
  not repeatedly scan the same huge folder.
- The database should track jobs, manifests, shards, leases, counters, and
  failures. It should not need one database row per PDF for the first scalable
  version.
- Work assignment should be dynamic. Faster servers claim more shards; failed
  or stopped servers release work through lease expiration.
- Output paths must be deterministic and derived from the manifest item, so
  retries do not create duplicate or conflicting outputs.

## Input Modes

### Folder Snapshot

The user selects:

```text
input_root = /shared/big_task
output_root = /shared/output/big_task
```

The platform recursively scans `input_root` once and creates a manifest:

```text
/shared/manifests/job-xxx/manifest.jsonl
/shared/manifests/job-xxx/manifest.meta.json
/shared/manifests/job-xxx/shards/shard-000001.jsonl
/shared/manifests/job-xxx/shards/shard-000002.jsonl
```

This job will always process the files recorded in that manifest, even if the
source folder changes later.

### Existing Manifest

The user provides a manifest path:

```text
manifest_path = /shared/manifests/prebuilt/manifest.jsonl
```

The platform validates the manifest format, records metadata, optionally shards
it if shard files do not already exist, and starts processing without scanning
the source folder.

This mode is preferred for extremely large jobs when the file list has already
been generated by upstream tooling.

### Delta Scan

Delta scan is a later phase. It compares a current folder scan against a prior
manifest and creates a new manifest containing added or changed PDFs.

Delta manifests should be new immutable inputs, not in-place edits of old
manifests.

## Manifest Format

Each manifest item is one PDF input.

Example `manifest.jsonl`:

```jsonl
{"input_path":"/shared/big_task/a/001.pdf","relative_path":"a/001.pdf","size_bytes":12345,"mtime_ns":1770000000000}
{"input_path":"/shared/big_task/b/inner/001.pdf","relative_path":"b/inner/001.pdf","size_bytes":45678,"mtime_ns":1770000001000}
```

Recommended fields:

- `input_path`: absolute path visible to all execution servers;
- `relative_path`: path relative to `input_root`;
- `size_bytes`: file size at scan time;
- `mtime_ns`: file modification time at scan time;
- `content_hash`: optional, expensive but stronger than size and mtime;
- `page_count`: optional, useful for better shard balancing if available;
- `source_id`: optional upstream id for audit or reconciliation.

Example `manifest.meta.json`:

```json
{
  "job_id": "job-xxx",
  "manifest_id": "manifest-xxx-v1",
  "input_mode": "folder_snapshot",
  "input_root": "/shared/big_task",
  "created_at": "2026-05-25T14:30:00Z",
  "scanner_version": "1",
  "file_count": 12345678,
  "total_bytes": 987654321000,
  "scanned_dir_count": 456789,
  "skipped_error_count": 17,
  "skipped_errors": [
    {
      "path": "/shared/big_task/blocked",
      "reason": "permission denied",
      "failure_category": "input_invalid"
    }
  ],
  "scan_started_at": "2026-05-25T14:00:00Z",
  "scan_finished_at": "2026-05-25T14:30:00Z",
  "options": {
    "recursive": true,
    "include_globs": ["**/*.pdf"],
    "exclude_globs": [],
    "follow_symlinks": false
  }
}
```

`scanned_dir_count` is the number of directories the scanner opened while
building the snapshot. `skipped_error_count` is the total number of scan paths
skipped because of permission, disappearance, or stat errors. `skipped_errors`
is a bounded sample for triage; it is not intended to hold every scan error in a
large production tree.

## Nested Folder Handling

Nested folders should not be assigned as scheduling units.

Bad approach:

```text
agent-01 gets /shared/big_task/a
agent-02 gets /shared/big_task/b
```

This creates uneven work when subfolders differ in file count, page count, or
OCR difficulty.

Preferred approach:

```text
scan recursively -> flatten to manifest items -> split into shards
```

The directory structure is only used for output mapping:

```text
input:
/shared/big_task/b/inner/001.pdf

relative_path:
b/inner/001.pdf

output:
/shared/output/job-xxx/b/inner/001.md
```

If duplicate basenames exist in different folders, preserving `relative_path`
prevents output collisions.

## Manifest Freshness

A manifest is immutable once a job starts. It represents the folder state at scan
time.

At execution time, each agent should perform a lightweight freshness check
before processing a PDF:

- file exists;
- size still matches `size_bytes`;
- mtime still matches `mtime_ns`.

If the file changed or disappeared, the agent should not silently process it.
It should report a structured failure:

- `input_missing`: file no longer exists;
- `input_changed`: size or mtime no longer matches;
- `input_invalid`: file exists but cannot be opened as a PDF.

This makes stale manifests observable instead of surprising.

## Shard Format

A shard is a bounded subset of manifest items.

Example:

```text
/shared/manifests/job-xxx/shards/shard-000123.jsonl
```

Each shard file can reuse the same JSONL item format as the full manifest.

Initial shard sizing should be configurable by file count:

```text
target_files_per_shard = 1000
```

Later, shard sizing can use estimated pages or bytes:

```text
target_pages_per_shard = 5000
target_bytes_per_shard = 10GB
```

For highly variable PDFs, smaller shards are safer because they improve dynamic
load balancing and reduce retry blast radius.

## Database Model

The first scalable version should add these concepts without storing every PDF
as a row.

### `manifests`

Suggested fields:

- `id`;
- `job_id`;
- `input_mode`;
- `input_root`;
- `manifest_path`;
- `meta_path`;
- `file_count`;
- `total_bytes`;
- `created_at`;
- `scanner_version`;
- `status`;
- `frozen_at`;
- `freeze_report_json`.

`next_shard_index` is a monotonic allocation cursor for distributed scans. When
a scan unit finishes, the control plane locks the manifest row, allocates global
shard indexes from `max(next_shard_index, existing max shard_index + 1)`, and
then advances `next_shard_index`. Unique database indexes on
`(job_id, shard_index)` and `(manifest_id, shard_index)` are still required as
the final guard. The job-level unique index prevents two distributed scan units
from registering the same global shard number even if they write different
manifest fragments. This keeps concurrent scan-unit completion safe on
PostgreSQL and also lets upgraded or manually repaired manifests recover if
`next_shard_index` lags behind existing shards.

`frozen_at` and `freeze_report_json` are written once distributed scanning has
no pending, running, or stale scan units and no failed scan units. The freeze
report is the audit snapshot for the scan phase: it records final file count,
total bytes, shard count, total files declared by shard rows, whether shard file
counts match the manifest file count, scan-unit counts, shard status counts, and
sampled scan errors. `scan_error_count` includes both failed scan units and
skipped errors reported by successful scan progress events, so a permissions
subtree is not hidden just because the scan unit still completed. This is
different from a live integrity check. The freeze report answers "what did the
platform believe was ready when scanning completed?" whereas integrity checks
answer "do the manifest and shard files still exist and match the database right
now?"

For distributed scans, the control plane may not have a merged global
`manifest.jsonl` mounted locally. In that mode, scan-unit manifest files are the
authoritative manifest fragments. The live integrity check parses JSONL rows
while validating each succeeded scan unit's `manifest_path` row count against
the database, validates its metadata file when declared, and then validates all
registered shard files the same way. Top-level manifest read/parse failures are
reported as `manifest_error`, top-level metadata JSON failures as `meta_error`,
top-level metadata `file_count` mismatches as `meta_file_count_matches`,
scan-unit metadata JSON failures as `meta_file_malformed` or
`meta_file_unreadable`, and scan-unit and shard parse failures as
`malformed_jsonl` issues. Malformed JSONL rows fail the report even when the
line count matches. It also compares the sum of registered shard `file_count` values with the
manifest or scan-unit manifest total, so a missing shard database row cannot
pass merely because the remaining shard files are self-consistent.
This lets a distributed scan be considered intact even when `manifest_path` is a
logical aggregate path rather than a physical file visible to the control host.

### `work_shards`

Suggested fields:

- `id`;
- `job_id`;
- `manifest_id`;
- `shard_index`;
- `shard_path`;
- `status`: `pending`, `running`, `succeeded`, `failed`, `stopped`;
- `assigned_server_id`;
- `attempt_count`;
- `lease_expires_at`;
- `started_at`;
- `finished_at`;
- `file_count`;
- `processed_files`;
- `failed_files`;
- `skipped_files`;
- `completed_pages`;
- `failure_category`;
- `error_message`.

### `shard_events`

Suggested fields:

- `id`;
- `job_id`;
- `shard_id`;
- `server_id`;
- `event_type`;
- `payload_json`;
- `created_at`.

The existing `job_events` table can still receive aggregated job-level events.
Shard events give enough evidence for retries and debugging without forcing the
default UI to load every file.

## Lease and Claiming

Agents should claim work dynamically:

```text
POST /api/agents/{server_id}/claim-shard
```

The control plane atomically selects a pending or expired shard and returns it:

```json
{
  "job_id": "job-xxx",
  "shard_id": "shard-000123",
  "shard_path": "/shared/manifests/job-xxx/shards/shard-000123.jsonl",
  "lease_expires_at": "2026-05-25T15:05:00Z"
}
```

While processing, the agent renews the lease:

```text
POST /api/shards/{shard_id}/heartbeat
```

If the agent dies or loses network connectivity, the lease expires. The shard
can then be claimed by another server.

Lease state should be based on database time, not agent-local clock time.

## Shard State Machine

```text
pending -> running -> succeeded
pending -> running -> failed
pending -> running -> stopped
running -> pending   # lease expired and retry is allowed
failed  -> pending   # explicit retry
```

Rules:

- A shard should only be assigned to one active server at a time.
- A stale `running` shard can be reclaimed after `lease_expires_at`.
- A shard should have a maximum retry count.
- Retry attempts should keep prior evidence.
- A job succeeds only when all shards are terminal and no shard is failed unless
  the job policy explicitly allows partial success.

## Agent Execution

For each claimed shard, the agent should:

1. Read the shard JSONL.
2. For each item, validate `input_path`, `size_bytes`, and `mtime_ns`.
3. Process PDFs with the configured OCR engine and model endpoint.
4. Derive output paths from `output_root` and `relative_path`.
5. Write outputs to shared storage.
6. Emit periodic shard progress events and renew the lease.
7. Report final shard status.

The current parser can initially run once per shard by giving it a shard-specific
input list or generated temporary input directory. A cleaner later version should
teach the parser to accept a manifest/shard file directly.

## Output Mapping

Output should be deterministic:

```text
output_root / relative_path-with-new-extension
```

Example:

```text
input_path:    /shared/big_task/b/inner/001.pdf
relative_path: b/inner/001.pdf
output_root:   /shared/output/job-xxx
markdown:      /shared/output/job-xxx/b/inner/001.md
json:          /shared/output/job-xxx/b/inner/001.json
assets:        /shared/output/job-xxx/b/inner/001_assets/
```

If the current parser creates output folders by basename, the shard execution
adapter must preserve enough relative path information to avoid collisions.
Manifest/shard execution must not enable `--flatten_output`; flattening removes
the `relative_path` parent directories and can merge distinct PDFs with the same
basename into one output location.
External manifests must also keep `relative_path` unique. Duplicate
`relative_path` rows map to the same output key and should be rejected before
OCR starts. If the control plane can read an external manifest during
registration, it should perform this check before creating shard metadata; if
the path is only visible to workers, the same invariant must be checked by the
parser and output audit tooling.

Shard reruns may skip an input only when the PDF output directory contains a
successful `.ocr_status.json`, that sidecar records `input_size_bytes` and
`input_mtime_ns` matching the manifest row, and all artifacts declared by that
sidecar still exist and are non-empty. Legacy Markdown files without a sidecar,
and older sidecars without input snapshot fields, are treated as incomplete in
manifest mode because they do not prove that the previous shard attempt handled
the same manifest snapshot. If the guard fails, manifest execution reprocesses
the PDF after the current input file passes the manifest freshness check. Output
audit reports missing sidecar input snapshot fields as `sidecar_input_missing`
and mismatches as `sidecar_input_mismatch`.
When manifest freshness validation fails before OCR starts, the failure sidecar
records both the current input `input_size_bytes`/`input_mtime_ns` and the
manifest row's `manifest_input_size_bytes`/`manifest_input_mtime_ns`, making
input drift diagnosable from the per-PDF output directory alone.

## UI Model

The default UI should show job and shard aggregates:

- total shards;
- pending/running/succeeded/failed/stopped shard counts;
- total PDFs from manifest metadata;
- processed PDF count;
- failed PDF count;
- completed pages;
- current throughput;
- active servers;
- stale shard count;
- failure category summary.

The default UI should not list every PDF.

Drill-down views should be bounded:

- recent failed shards;
- recent failed PDFs sampled from shard outputs or failure sidecar files;
- active shards;
- stale shards;
- exact input path search.

## Failure Categories

Shard-level execution should reuse the platform failure taxonomy:

- `input_missing`;
- `input_changed`;
- `input_invalid`;
- `output_unwritable`;
- `model_unreachable`;
- `model_timeout`;
- `model_error`;
- `model_output_invalid`;
- `postprocess_failed`;
- `artifact_missing`;
- `operator_stopped`;
- `runner_start_failed`;
- `runner_lost`;
- `parser_failed`;
- `process_failed`;
- `process_killed`;
- `unknown`.

The job summary should aggregate these categories without needing to scan every
item in the manifest.

## Recommended Implementation Phases

### Phase 1: Manifest Snapshot

- Add folder scan endpoint or background scanner.
- Generate `manifest.jsonl` and `manifest.meta.json`.
- Preserve nested folder `relative_path`.
- Support existing manifest input mode.
- Do not change multi-server execution yet.

### Phase 2: Static Shards

- Split manifest into shard JSONL files.
- Add `manifests` and `work_shards` tables.
- Show shard counts in the UI.
- Run a single-server job shard by shard to validate output mapping.

### Phase 3: Dynamic Claiming

- Add claim and heartbeat APIs.
- Let multiple agents claim shards dynamically.
- Implement lease expiration and reclaim.
- Keep job-level UI aggregate-only.

### Phase 4: Retry and Recovery

- Add max retry policy.
- Add stale shard recovery.
- Preserve attempt evidence.
- Add bounded recent failure APIs.

### Phase 5: Delta Scan

- Compare a new scan against an old manifest.
- Generate delta manifests for added or changed PDFs.
- Treat delta manifests as new immutable job inputs.

## Open Questions

- Should a shard be sized by file count first, or should we invest early in page
  count estimation?
- Should `input_changed` be a hard failure by default, or should operators be
  allowed to process changed files with an explicit override?
- Should output completeness be checked per PDF, per shard, or both?
- How should duplicate `relative_path` entries be handled if an external
  manifest is malformed?
- Should scanner run inside the control server, as a separate scanner process,
  or as a special agent role?

## Recommended Next Step

Implement Phase 1 and Phase 2 together as the next architectural milestone:

1. Add manifest generation for a recursive folder snapshot.
2. Add existing manifest input mode.
3. Add shard file generation.
4. Add DB rows for manifest and shard metadata.
5. Keep execution single-agent at first, but make the data model ready for
   dynamic multi-agent claiming.

This keeps the implementation testable while moving the platform toward the
10-server shared-disk production model.
